## 三、已实施的优化

### P0: 1m K线方向感知过滤

**位置**: `deep_alpha_pro.py:2963-2983`

**问题**: 原有 `body > 15%` 无差别杀死大阳线和大阴线，会误杀 Top20 中 7 个大奖（含 WOMBLE +1471%）。

**改动**:
```python
# 大阴线 body>10% + SM=0 → 跳过（砸盘无人接）
# 阴线 body>5% + SM=0 → 跳过（阴跌无支撑）  
# 上影线 >8% → 跳过（抛压重）
# 大阳线（绿盘）→ 始终放行
# 有 SM 的红盘 → 放行（聪明钱可能在吸筹）
```

**效果**: 
- Top20 被误杀: 7 → 2
- 盈利盘被误杀: 0
- 大阳线突破不再被拦截

---

### P1: 20-50K 弱筹码过滤

**位置**: `deep_alpha_pro.py:2985-2991`

**问题**: 20-50K 区间中，SM<3 且 Top10>20% 的组合胜率 0%，死率 57%。

**改动**:
```python
if 20000 <= mcap < 50000 and sm_count < 3 and top10_rate > 20:
    continue  # 弱筹码结构
```

**效果**: 砍掉 8 个无效信号，0 个盈利盘误杀。

---

### P2: 20-30K 无 SM 过滤

**位置**: `deep_alpha_pro.py:2992-2995`

**问题**: 20-30K + SM=0 的子集 100% 死亡。

**改动**:
```python
if 20000 <= mcap < 30000 and sm_count < 1:
    continue  # 无聪明钱站台
```

**效果**: 砍掉 2 个无效信号，0 误杀。

---

### P3: 推送后 1m K线持续跟踪

**位置**: `deep_alpha_pro.py:2554-2757`, Redis 函数 `661-708`, 配置 `88-103`

**问题**: 推送后无跟踪，冲高回落盘（48.5%）用户不知道何时退出。

**机制**:
- 推送成功后注册到 Redis (`start_tracking`)
- 每轮 `scan_pro()` 末尾检查所有跟踪中的代币 (`check_tracked_tokens`)
- 频率控制：前 15m 每轮检查，15-30m 隔轮，30-60m 隔 5 轮
- 拉取 1m K线，判断告警条件
- 60 分钟后发终检总结，自动清理

**告警条件**（每个只触发一次）:

| 条件 | 消息 |
|------|------|
| 现价 >= entry × 5 | 🚀 5x达成 |
| 现价 >= entry × 2 | 💰 已翻倍 |
| 现价 >= entry × 1.5 | 📈 +50% |
| 从高点回落 40% | ⚠️ 高点回落40% |
| 从高点回落 20% | 🔻 高点回落20% |
| 跌破入场 + 回落 15% | 🔴 跌破入场价 |

**关闭方式**: 环境变量 `DEEP_ALPHA_TRACK_ENABLED=0`

## 四、完整过滤链路

```
GMGN 热门榜 (top 100, 1m)
  │
  ├─ 市值 < $7K 或 > $1M → 跳过
  ├─ 3 轮价格观察不达标 → 跳过
  ├─ 深度分析: dev 买入 > $500 或持仓 > 30% → 跳过
  ├─ 创建 > 24h → 跳过
  ├─ 手续费 < 1 SOL → 跳过
  ├─ 高交易量低手续费（反刷量）→ 跳过
  ├─ 砸盘进行中 → 跳过
  │
  ├─ [P0] 1m K线方向过滤
  │   ├─ 大阴线 body>10% + SM=0 → 跳过
  │   ├─ 阴线 body>5% + SM=0 → 跳过
  │   ├─ 上影线 >8% → 跳过
  │   └─ 量比 < 0.3 → 跳过
  │
  ├─ [P1] 20-50K + SM<3 + Top10>20% → 跳过
  ├─ [P2] 20-30K + SM=0 → 跳过
  │
  └─ ✅ 推送消息
       └─ [P3] start_tracking() → Redis
            └─ 每轮 scan 检查 → 60m 终检自动清理
```
